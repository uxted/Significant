# Specification: Система агрегирования и визуализации экономически значимых новостей

> **Тема ВКР:** Разработка программного комплекса агрегирования и визуализации экономически значимых новостей на основе методов машинного обучения
>
> **Срок сдачи:** 17 апреля 2026 г.
>
> **Технологический стек:** Python, Pandas, scikit-learn, TensorFlow, BeautifulSoup, React, Django, PostgreSQL, Redis, Celery, Docker

---

## 1. Что делаем (In Scope)

### 1.1 Сбор новостей

- [ ] Парсинг RSS-лент (XML-фиды)
- [ ] Прямой парсинг веб-страниц (BeautifulSoup)
- [ ] ~15 источников (руссязычные + ключевые международные события в русскоязычном изложении)
- [ ] Источники 4-х уровней: официальные регуляторы, информагентства, деловые СМИ, корпоративные раскрытия
- [ ] Автоматический опрос каждые 5 минут (Celery Beat)
- [ ] Дедупликация по хешу заголовка (окно 24 часа)

### 1.2 Классификация новостей

- [ ] Двухуровневая система:
  - **Уровень 1:** Бинарная ML-классификация (значимая=1 / не значимая=0)
  - **Уровень 2:** Постобработка — присвоение приоритета High/Medium/Low по правилам
- [ ] Rule-based фильтр (ключевые слова + проверка источника, отсеивает 60-70% явно не значимых)
- [ ] RuBERT fine-tuned как основная модель
- [ ] TF-IDF + Logistic Regression как fallback
- [ ] Иерархия источников влияет на вес новости и порог классификации

### 1.3 Веб-интерфейс

- [ ] Лента новостей с цветовой маркировкой (🔴 High / 🟡 Medium / 🟢 Low)
- [ ] Сводная статистика за сегодня (количество по уровням и категориям)
- [ ] Панель фильтров: дата (диапазон), значимость (H/M/L), категория, источник, полнотекстовый поиск
- [ ] Карточка новости: заголовок, источник, дата, уровень значимости, категория, краткое описание, затрагиваемые активы (тикеры), ссылка на оригинал
- [ ] Детальный просмотр: полный текст новости
- [ ] Пагинация: page-based (page/page_size, max 100)
- [ ] Polling для автообновления ленты (30 сек активная вкладка, 120 сек неактивная)
- [ ] Регистрация/вход (email + пароль → JWT)
- [ ] Подписки на компании/категории
- [ ] Закладки новостей
- [ ] Админ-панель: управление источниками, мониторинг парсеров, ручная модерация, audit log

### 1.4 API

- [ ] REST API (JSON) через Django REST Framework
- [ ] JWT-аутентификация (access 15 мин + refresh 7 дней, djangorestframework-simplejwt)
- [ ] Rate limiting: 100 запросов/мин (пользователь), 30/мин (аноним), 500/мин (админ)
- [ ] Основные endpoints: `/api/news/`, `/api/news/{id}/`, `/api/sources/`, `/api/categories/`, `/api/user/subscriptions/`, `/api/user/bookmarks/`, `/api/admin/sources/`, `/api/admin/moderation/`, `/api/auth/login/`, `/api/auth/refresh/`

### 1.5 Тестирование

- [ ] Unit-тесты бэкенда (pytest, покрытие ≥80%)
- [ ] Unit-тесты фронтенда (Jest, покрытие ≥70%)
- [ ] E2E-тесты критических сценариев (Playwright, 3-5 сценариев)
- [ ] Нагрузочное тестирование API (Locust, 10-20 пользователей)
- [ ] Тестирование ML-модели (5-fold CV, hold-out набор 200 новостей)
- [ ] Логирование метрик (Confusion matrix, ROC-кривая, Precision-Recall curve)
- [ ] CI/CD pipeline (GitHub Actions)

### 1.6 Безопасность

- [ ] Хранение паролей: Django PBKDF2 (SHA256, 600 000 итераций)
- [ ] Rate limiting через Redis
- [ ] Блокировка на 15 минут после 5 неудачных попыток входа
- [ ] Логирование действий администратора (audit log, 180 дней)
- [ ] Логирование аутентификации (90 дней)
- [ ] Чекбокс согласия на обработку ПДн при регистрации
- [ ] Страница политики конфиденциальности (`/privacy/`)
- [ ] Кнопка удаления аккаунта пользователя
- [ ] Секреты в `.env` (не коммитится в Git)

## 4. Риски и митигация

| № | Риск | Вероятность | Влияние | Митигация |
|---|------|-------------|---------|-----------|
| R1 | ML-модель не достигает Precision ≥80% | Средняя | Высокое | Дообучение на дополнительных данных; корректировка порога классификации; использование ансамбля (RuBERT + TF-IDF) |
| R2 | Источники меняют структуру сайтов → парсеры ломаются | Высокая | Среднее | Логирование ошибок парсинга; уведомления администратору; модульная архитектура парсеров (легко обновить) |
| R3 | Недостаточно RAM для всех контейнеров | Средняя | Высокое | Уменьшение количества Celery workers до 1; отключение неиспользуемых сервисов; рекомендация 16 GB RAM |
| R4 | Датасет недостаточного качества (разметка) | Средняя | Высокое | Проверка разметки экспертом (руководитель ВКР); weak supervision для расширения |
| R5 | Docker не запускается на компьютере в аудитории | Низкая | Высокое | Скринкаст как резерв; подробная инструкция в README; проверка Docker заранее |
| R6 | Защита перенесена / чужой компьютер | Низкая | Высокое | Скринкаст 5 минут; портативность Docker Compose; наличие резервной копии на USB |
| R7 | Переполнение диска логами | Низкая | Среднее | Лог-ротация; автоматическая очистка старше 30 дней |
| R8 | Утечка .env файла | Низкая | Высокое | Немедленная ротация SECRET_KEY и JWT_SIGNING_KEY; все токены инвалидируются |
| R9 | Превышение времени обработки (>1 сек) | Средняя | Среднее | Оптимизация запросов; добавление индексов БД; кэширование Redis |
| R10 | Падение PostgreSQL | Низкая | Высокое | Автоматический рестарт через Docker; ежедневный бэкап (Celery задача в 04:00) |
| R11 | Потеря модели | Низкая | Высокое | Резервная копия в Git LFS или облачном хранилище (Google Disk) |
| R12 | Конфликты портов на демонстрационном компьютере | Средняя | Среднее | Настройка переменных окружения для смены портов; документация |
| R13 | Дубликаты новостей из разных источников | Высокая | Низкое | Дедупликация по хешу заголовка + окно 24 часа; группировка в кластеры |
| R14 | Противоречивые данные из разных источников | Средняя | Среднее | Приоритет источнику уровня 1; метка «требует проверки» для новостей с confidence 0.4-0.6 |
| R15 | Тесты не проходят в CI | Средняя | Среднее | Блокировка merge до исправления; уведомление разработчика |

## 5. Тест-подход

### 5.1 ML-модель

| Аспект | Детали |
|--------|--------|
| **Датасет** | 5000 новостей (2500 значимых, 2500 не значимых) |
| **Разделение** | Train 70% / Validation 15% / Test 15% (750 новостей) |
| **Кросс-валидация** | 5-fold CV для подбора гиперпараметров |
| **Hold-out** | Отдельные 200 новостей (финальная проверка) |
| **Baseline** | Rule-based (~60% Precision), TF-IDF + Logistic Regression (~70%) |
| **Целевые метрики** | Precision ≥80%, Recall ≥75%, F1 ≥0.77 |
| **Артефакты** | Confusion matrix (PNG), ROC-кривая (PNG), Precision-Recall curve (PNG), report.json |
| **Логирование** | Время инференса, confidence score (агрегированно) |

### 5.2 Бэкенд (Django)

| Тип | Инструменты | Покрытие |
|-----|-------------|----------|
| Unit-тесты | pytest, DRF test client, factory_boy | ≥80% |
| Integration-тесты | pytest + Celery + БД | Критические пути ≥95% |
| Компоненты | Парсинг (mock RSS), классификация (mock ML), API endpoints, аутентификация, модели БД, Celery задачи | ≥70-90% по модулям |
| Сценарии | Полный цикл парсинга, аутентификация + доступ, фильтрация + пагинация, дедупликация | — |

### 5.3 Фронтенд (React)

| Тип | Инструменты | Покрытие |
|-----|-------------|----------|
| Unit-тесты | Jest, React Testing Library | ≥70% |
| Компоненты | NewsCard, NewsFeed, Filters, LoginForm, хуки (useNewsFeed, useAuth) | — |
| E2E-тесты | Playwright | 3-5 критических сценариев |
| E2E-сценарии | Регистрация → Вход → Лента; Применение фильтров; Поиск; Закладки | — |

### 5.4 Производительность

| Тест | Инструмент | Параметры | Критерий |
|------|-----------|-----------|----------|
| Нагрузка API | Locust | 10-20 пользователей, 5 минут | 95-й перцентиль ≤500 мс |
| Время обработки новости | Логирование Celery | Полный цикл (парсинг → ML → БД) | ≤1 секунды |
| Время инференса ML | Логирование | RuBERT prediction | ≤0.5 секунды |
| Масштабирование БД | Ручной тест | 1K / 10K / 50K записей | ≤500 мс при 50K |

### 5.5 CI/CD (GitHub Actions)

```
on: [push, pull_request]
  ↓
Install dependencies (Python + Node.js)
  ↓
Linting: Black + Flake8 (Python), ESLint + Prettier (JS)
  ↓
Unit tests: pytest (backend), Jest (frontend)
  ↓
ML metrics validation (Precision ≥80%, Recall ≥75%)
  ↓
Build Docker images
```

## 7. Критерии экономической значимости

### 7.1 Классификация

- **ML-модель:** бинарная (значимая=1 / не значимая=0)
- **Post-processing:** 3 уровня (High/Medium/Low) по правилам на основе типа события и источника

### 7.2 Признаки значимости

| Категория | Конкретные признаки |
|-----------|---------------------|
| Макроэкономика | Изменение ключевой ставки, инфляция, ВВП, безработица |
| Корпоративные события | Отчётность (выручка >75 млрд руб.), дивиденды, M&A, IPO, смена CEO |
| Регуляторные решения | Законы, санкции, лицензирование, изменения в налогообложении |
| Рыночные события | Листинг/делистинг, остановка торгов, дефолт, реструктуризация |
| Геополитика | События, влияющие на торговлю, логистику, доступ к технологиям |

**Количественные пороги:**
- Выручка компании > 75 млрд руб.
- Активы компании > 150 млрд руб.
- Сумма сделки M&A > 10 млрд руб.

### 7.3 Иерархия источников

| Уровень | Тип источника | Примеры | Влияние на классификацию |
|---------|--------------|---------|-------------------------|
| 1 (макс.) | Официальные регуляторы и эмитенты | cbr.ru, e-disclosure.ru, government.ru | Авто-значимость или prior=0.9 для ML |
| 2 | Крупные информагентства | ТАСС, РИА, Интерфакс, РБК | Высокий вес, порог классификации снижен |
| 3 | Деловые СМИ и отраслевые издания | Коммерсантъ, Ведомости, Smart-Lab | Стандартная классификация по модели |
| 4 | Корпоративные раскрытия | e-disclosure.ru, сайты эмитентов | Стандартная классификация |

## 8. Граничные случаи (Edge Cases)

| Случай | Обработка |
|--------|-----------|
| Противоречивые данные из разных источников | Приоритет источнику уровня 1, метка «требует проверки» |
| Слух без подтверждения | Пометка «неподтверждено», не показывать в топе ленты |
| Новости на стыке категорий (спорт + экономика) | Классификация по ML-модели, при confidence <0.6 → ручная модерация |
| Повторная публикация одной новости | Дедупликация по хешу заголовка + времени (окно 24 часа) |
| Изменение статуса новости (подтверждение слуха) | Автоматическое обновление метки при появлении источника уровня 1-2 |
| Пользователь не настроил подписки | Показывать все значимые новости по умолчанию |
| Пустая выдача по фильтрам | Сообщение «Нет новостей за выбранный период», предложение расширить фильтры |
| Новость обновлена после публикации | Метка «Обновлено», сохранение истории изменений |
| Дубликаты одной новости из разных источников | Группировка в кластер, показ источника-лидера |
| Ошибка загрузки ленты | Кэшированная версия + сообщение «Попытка повторного подключения» |
| API недоступен во время polling | Повторная попытка через 10 секунд, уведомление пользователя |
| Токен истёк во время запроса | Автоматический refresh, повтор запроса |
| ML-модель недоступна (ошибка загрузки) | Queue на повтор, новость помечается «требует обработки» |
| Превышение лимита запросов (rate limit) | Возврат 429, polling приостанавливается на 60 секунд |
| Изменение структуры сайта-источника | Логирование ошибки, уведомление администратора |
| Потеря соединения с БД | Celery задача перезапускается через 5 минут |
| 5+ неудачных попыток входа | Блокировка на 15 минут, логирование WARNING |
| Пользователь запросил удаление аккаунта | Анонимизация ПДн в течение 24 часов, сохранение новостей без связи с user_id |
| SQL-инъекция через поиск | Django ORM экранирует параметры, дополнительный валидатор ввода |
| Попытка доступа к чужим закладкам | Проверка user_id на уровне API (owner_only permission) |
